# Goal

在 REQ-033 统一 Envelope 基础上，引入消息生命周期目录结构，
实现 Claim 原子性机制，防止 cron 重启或多消费者场景下的重复消费。

**这是防止 Agent 任务被重复触发的核心防护机制。**

# In Scope

- `scripts/pandas-heartbeat.sh`：`inbox_init()` 创建四级子目录（pending/claimed/done/failed）
- `scripts/pandas-heartbeat.sh`：`inbox_write_v2()` 默认写入 `for-{target}/pending/`
- `scripts/pandas-heartbeat.sh`：`inbox_read_pandas()` 实现 Claim 原子性
  1. 扫描 `for-pandas/pending/*.md`
  2. `mv pending/$f claimed/$f`（原子性；mv 失败则 skip）
  3. 执行 handler 处理
  4. 处理成功：`mv claimed/$f done/$f`
  5. handler 异常退出：`mv claimed/$f failed/$f`，追加错误摘要到文件末尾
- 旧扁平目录兼容：残存旧格式文件仍可被读取处理

# Out of Scope

- 文件命名规范变更 → REQ-036
- Thread/Correlation 链路追踪 → REQ-035
- expired 状态 / watchdog 超时检测

# Acceptance Criteria

- `inbox_init()` 创建 `for-pandas/{pending,claimed,done,failed}/` 及 menglan / huahua 对应目录
- `inbox_write_v2()` 写入 `for-{target}/pending/`（而非扁平目录）
- `inbox_read_pandas()` 在读取每条消息前先 `mv pending/$f claimed/$f`
- 若 mv 失败（文件已不存在），直接 skip（不报错）
- handler 成功后 `mv claimed/$f done/$f`
- handler 失败后 `mv claimed/$f failed/$f` + 追加错误摘要到文件末尾
- `npm run build` 通过
- `npm run release:audit` 通过
- `npm test` 通过，新增 Claim 机制单元测试

# Test Case Design Notes

需覆盖以下场景：

- TC-034-01：正常 claim — mv 成功 → handler 执行 → done/
- TC-034-02：Claim 竞争 — 文件已被 mv（模拟），mv 失败 → skip，不报错
- TC-034-03：Handler 成功 — 消息移到 done/，文件内容与原 pending/ 一致
- TC-034-04：Handler 失败 — 消息移到 failed/，文件末尾有错误摘要
- TC-034-05：inbox_init 幂等 — 重复调用不报错，目录已存在时不覆盖
- TC-034-06：inbox_write_v2 落点 — 新写入消息在 pending/ 而非扁平目录
- TC-034-07：旧扁平目录兼容 — 残存旧格式文件仍可被处理
- TC-034-08：failed/ 文件内容 — 包含原 frontmatter + 末尾错误摘要行
