# REQ-037 — git worktree 隔离

## 背景

当前三个 agent（Pandas / Menglan / Huahua）共享同一个本地 git working tree
（`~/workspace-pandas/open-workhorse`）。协议层通过顺序消息传递（inbox → heartbeat →
result）假定同一时间只有一个 agent 写 repo，但这个保证是软性的：

- cron 时间错开 / stall watchdog 误判 / 手动触发均可导致两个 agent 同时持有写锁
- Menglan 在 feature branch 有 unstaged changes 时，Pandas 心跳执行 `git checkout main`
  会报错或意外覆盖
- git index 是进程共享状态，任何 `git add` / `git commit` 都可能踩到对方

## 方案

使用 **git worktrees** 做物理隔离：

```
主 checkout（main）      ~/workspace-pandas/open-workhorse/    ← Pandas 操作
Menglan worktree        ~/workspace-menglan/open-workhorse/   ← feature branch
Huahua                  只读 PR diff（gh pr diff），不需要本地 checkout
```

每次 `harness.sh implement <REQ-N>` 时：
1. 在主 repo 创建 worktree 并绑定 `feat/REQ-N` 分支
2. 把 Menglan 的工作目录指向该 worktree
3. Menglan 在独立文件系统视图内完成实现、测试、push
4. PR merge 后执行 `git worktree remove` 清理

## Acceptance Criteria

- [ ] `harness.sh implement <REQ-N>` 自动创建 worktree 至 `~/workspace-menglan/open-workhorse/`
- [ ] worktree 绑定 `feat/REQ-N` 分支（分支不存在则自动创建）
- [ ] worktree 已存在时幂等（不重复创建）
- [ ] `harness.sh worktree-clean <REQ-N>` 或 PR merge 后自动执行 `git worktree remove`
- [ ] Pandas 在主 checkout（main）的操作不受 worktree 状态影响
- [ ] `npm run release:audit && npm run build && npm test` 全部通过

## 实现范围

- `scripts/harness.sh`：implement 命令加 worktree 创建逻辑
- `harness/agent-cli-playbook.md`：模板 B/K 更新，说明 Menglan 在 worktree 路径操作
- `harness/harness-index.md`：自动化流程说明更新

## 优先级说明

P1：当前顺序协议在单任务场景下可工作，但并发/误触场景下有数据损坏风险。
随着任务数量增加和 watchdog 更积极，冲突概率上升。
