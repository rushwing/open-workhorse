# Goal

实现 Daniel 与 Pandas 的双向 Telegram 通信：Daniel 可发送文本指令控制编排循环，
Pandas 可通过 `tg_decision` 阻塞等待 Daniel 决策。

设计细节见 `docs/PANDAS-ORCHESTRATION.md` §6。

# In Scope

- `scripts/telegram.sh` 新增函数：
  - `tg_poll_commands` — 调用 `getUpdates`，返回未处理的文本消息
  - `tg_decision <context> <options>` — 发送决策消息，poll 直到 Daniel 回复或超时（默认 24h）
  - `tg_pr_ready <url> <summary>` — 发送 PR 合并通知（[Merge] [Hold]）
- `scripts/pandas-heartbeat.sh` 在每次心跳时调用 `tg_poll_commands`，解析：
  - `start REQ-N` → 立即认领并分发 REQ-N
  - `status` → 扫描 tasks/ 并发送当前任务状态摘要
  - `hold` → 写标志文件 `.pandas_hold`，暂停 auto-dispatch
  - `resume` → 删除 `.pandas_hold`，恢复 auto-dispatch
- 已处理消息通过 `offset` 参数标记为已读（Telegram getUpdates 约定）

# Out of Scope

- Telegram 机器人创建和 token 申请（用户预先配置）
- 非文本消息（图片、文件等）的解析
- 多用户支持（仅 Daniel 的 TELEGRAM_CHAT_ID）

# Acceptance Criteria

- `tg_poll_commands` 在 `TELEGRAM_BOT_TOKEN` 和 `TELEGRAM_CHAT_ID` 未设置时静默跳过（非 fatal）
- `tg_decision` 发送消息后每 30s poll 一次，直到收到回复或超时
- `hold` 指令创建 `.pandas_hold` 文件；`resume` 删除之；auto-dispatch 检查该文件
- offset 更新正确（处理完 update_id=N 后，下次请求 offset=N+1）
- `bash -n scripts/telegram.sh` 无语法错误
- `bash -n scripts/pandas-heartbeat.sh` 无语法错误
- `npm run release:audit` 通过
- `npm run req:check` 通过

# Test Case Design Notes

- TC-024-01: TELEGRAM_BOT_TOKEN 未设置时，tg_poll_commands 静默返回（不报错）
- TC-024-02: mock getUpdates 返回 `start REQ-021`，断言 REQ-021 被认领
- TC-024-03: mock getUpdates 返回 `hold`，断言 `.pandas_hold` 文件存在
- TC-024-04: mock getUpdates 返回 `resume`，断言 `.pandas_hold` 文件已删除
- TC-024-05: tg_decision 超时（mock：poll 超过 max_wait）→ 重新发送告警

# Agent Notes

- Telegram getUpdates API: `GET /bot{token}/getUpdates?offset={offset}&timeout=0`
- 返回 JSON: `{ok: true, result: [{update_id, message: {chat: {id}, text}}]}`
- offset 持久化：写入 `.pandas_tg_offset` 文件（gitignore 该文件）
- 仅处理 `message.chat.id == TELEGRAM_CHAT_ID` 的消息（过滤其他来源）
- `tg_decision` 超时行为：超时后通过 `tg_notify` 再次告警，不自动决策
