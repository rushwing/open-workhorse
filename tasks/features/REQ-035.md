# Goal

在 REQ-033 Envelope 基础上，落实 Thread / Correlation 的创建与传播规则，
使一个 REQ 的完整协作链路（Pandas↔Menglan↔Huahua 多轮往返）可以被追踪和重建。

# In Scope

- `scripts/pandas-heartbeat.sh`：`auto_claim()` 在路由第一个 request 时创建 `thread_id`
  - 格式：`thread_{req_id}_{epoch}`
  - 持久化：**仅携带在消息 Envelope frontmatter 中**，不写回 REQ 文件（Daniel 决策 2026-03-20）
  - 可追踪性：消息进入 done/failed/ 后，通过 `grep thread_id done/` 可重建完整链路
- `scripts/pandas-heartbeat.sh`：每次发出新 request 复用已有 `thread_id`，创建新 `correlation_id`
- `scripts/pandas-heartbeat.sh`：处理 response 时，提取 `correlation_id`，确认与发出的 request 配对
  - 配对失败 → warn + 写入 `failed/`
- 新增辅助函数 `thread_get_or_create <req_id>`
- 新增辅助函数 `correlation_new <req_id>`

# Out of Scope

- UI 层 thread 链路可视化
- 跨 Team 的 thread

# Acceptance Criteria

- `auto_claim()` 路由第一个 request 时，生成 `thread_id: thread_{req_id}_{epoch}` 并写入消息 Envelope（不写回 REQ 文件）
- 后续同一 REQ 的所有 request 携带相同 `thread_id`，不同 `correlation_id`
- correlation_id 不匹配时 → warn 日志 + 消息移到 failed/
- `thread_get_or_create` 函数幂等
- `npm run build` 通过
- `npm run release:audit` 通过
- `npm test` 通过，覆盖 thread 创建、复用、correlation 校验场景

# Test Case Design Notes

需覆盖以下场景：

- TC-035-01：首次 request — thread_id 创建，写入消息 Envelope，不修改 REQ 文件
- TC-035-02：第二次 request（同一 REQ）— 复用已有 thread_id，新 correlation_id
- TC-035-03：response 配对 — correlation_id 匹配 → handler 正常执行
- TC-035-04：response correlation_id 不匹配 → warn + 消息移到 failed/
- TC-035-05：`thread_get_or_create` 幂等性 — 两次调用返回相同值
- TC-035-06：完整链路重建 — 给定 thread_id，在 done/ 中 grep 可找到所有相关消息
