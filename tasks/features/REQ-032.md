# Content Strategy

采用**方案1（绝对路径引用）**：

- 消息 body 只包含任务的绝对文件路径（e.g. `task_file: /home/openclaw/workspace-pandas/open-workhorse/tasks/features/REQ-032.md`）
- 接收方在处理时自行读取文件获取最新状态
- 好处：任务被人工处理后，接收方读到最新状态可自动感知无需处理；消息体积小；ground truth 永远是文件系统
- 坏处：多一次 Read，但成本可接受

# Filename Format

```
{YYYY-MM-DD}-{sender}-{type}-{work_item_id}.md

# 示例
2026-03-19-pandas-implement-REQ-032.md
2026-03-19-huahua-dev_complete-REQ-032.md
2026-03-19-menglan-major_decision_needed-REQ-032.md
```

- 去掉当前的 PID+RANDOM 后缀（同类型同 work_item 的重复发送本身就是问题，应由幂等检查处理，不应靠随机后缀绕过）
- LLM 读目录列表时可仅凭文件名做初步路由判断

# Test Case Design Notes

需覆盖场景：
- 合法 type 消息 → 正常路由处理
- 未知 type 消息进 dead-letter → for-pandas 收到告警
- 同 work_item 重复发送同 type 消息 → 幂等（第二条覆盖或跳过，不重复触发 Claude）
- task_file 路径对应任务已被处理（status=done）→ 接收方跳过，不重复执行
