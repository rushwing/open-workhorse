# Goal

修复并扩展 `pandas-heartbeat.sh` 中针对 umbrella REQ 的编排逻辑，实现：

1. Pandas 正确识别 umbrella REQ，**不**直接路由 implement 给 Menglan
2. 子 REQ 全部 `review` 后，Pandas 自动检测各子 REQ 对应 PR 是否 merged
3. PR 全部 merged 后，Pandas 批量将子 REQ 标记为 `done`
4. Pandas 路由 umbrella REQ 给 Menglan 做准出核查
5. Menglan 核查通过 → 标记 umbrella `done` + 通知 Daniel
6. Menglan 无法核查（准出条件未分解完全）→ dead-letter + `tg_notify` 告警上报

背景：当前 `dead-letter/2026-03-15-pandas-implement-req-020-130534-13052.md` 证明 umbrella REQ 被错误路由，本 REQ 修复该问题并补全端到端闭环。

# Agent Notes

## 关键设计决策

### PR 号如何获取

子 REQ 文件当前没有 `pr_number` 字段。检测方案（优先级顺序）：
1. `gh pr list --search "REQ-N in:title" --state merged` — 无需修改 REQ 文件
2. 若子 REQ 文件补充了 `pr_number` 字段，直接用 `gh pr view <num> --json state`

建议实现方案 1（无侵入），方案 2 作为 fallback。

### umbrella_review 消息格式

```yaml
---
type: umbrella_review
req_id: REQ-N
summary: 请核查 REQ-N umbrella 准出条件
status: blocked
---
（umbrella REQ 完整 spec 内容，含 Acceptance Criteria 原文）
```

### umbrella_complete 消息格式

```yaml
---
type: umbrella_complete
req_id: REQ-N
status: blocked
blocking_reason: ""   # 仅 blocked 时填写，说明哪条准出条件无法核查
---
```

### Dead-letter 检测

`pandas-heartbeat.sh` 在处理完 `for-pandas/` inbox 后，扫描 `dead-letter/` 目录：
- 若存在文件 → `tg_notify "⚠️ dead-letter: $(basename $f) — 请人工介入"` + 保留文件（不删除，避免重复告警可加 `.acked` 后缀）
