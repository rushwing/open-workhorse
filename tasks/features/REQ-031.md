# Goal

实现 PR merge 后的归档全自动化，消除当前 `tg_pr_ready` → Daniel merge 之后流程断裂的问题。

Pandas 心跳在检测到 `status: review` 的 REQ 对应 PR 已合并后，无需人工干预，自动完成：

1. REQ `status → done`
2. 关联 TC `status → done`
3. `mv tasks/features/REQ-xxx.md → tasks/archive/done/`
4. `mv tasks/test-cases/TC-xxx.md → tasks/archive/done/`
5. commit（使用 `ARCHIVE_WORK_ITEM_COMMIT` binding）
6. Telegram 通知 Daniel 归档完成

同时修复 BUG-005：归档时将 TC `status` 字段更新为 `done`（不只是移动文件）。

# In Scope

- `scripts/pandas-heartbeat.sh`：新增 `archive_merged_reqs()` 函数
  - 扫描 `tasks/features/` 中 `status: review` 的 REQ
  - 读取 frontmatter 中的 PR 编号（从 `Agent Notes` 或新增 `pr_number` 字段）
  - 调用 `gh pr view <pr_number> --json state` 判断是否已 merge
  - 已 merge → 更新 REQ status + TC status → mv → commit → tg_notify
- `requirement-standard.md §9.3`：补充 TC status 更新为 done 的要求（BUG-005 修复）
- 新增对应 TC 文档

# Out of Scope

- Daniel 确认步骤（本 REQ 实现全自动化，无需二次确认；如需确认门控另开 REQ）
- `review_ready → req_review` claim 逻辑（BUG-004 单独修复）
- 历史已归档但 status 不正确的 TC 批量修正（在本 PR 中附带修复）

# Acceptance Criteria

- `status: review` 的 REQ 在对应 PR merge 后，下一次 Pandas 心跳自动归档
- 归档后 `tasks/archive/done/REQ-xxx.md` 的 `status` 为 `done`
- 归档后 `tasks/archive/done/TC-xxx.md` 的 `status` 为 `done`（BUG-005）
- 归档 commit message 使用 `archive({WORK_ITEM_ID}): move to tasks/archive/done/` 格式
- Daniel 收到 Telegram 归档完成通知
- `npm test` 通过，覆盖新增 TC 场景
- `npm run release:audit && npm run build` 通过

# Test Case Design Notes

需覆盖场景：
- PR 已 merge → 触发归档
- PR 尚未 merge（open/closed）→ 不触发
- REQ 有多个关联 TC → 全部归档，status 全为 done
- gh 调用失败（网络/权限）→ 跳过该 REQ，不中断心跳，记录 warn
- 重复执行（REQ 已归档）→ 幂等，不报错
