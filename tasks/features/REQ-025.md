# Goal

实现重大决策检测机制：Pandas 在编排循环中自动识别预定义触发条件，
打断当前状态机，通过 `tg_decision` 向 Daniel 升级，等待决策后继续或暂停。

设计细节见 `docs/PANDAS-ORCHESTRATION.md` §5。

# In Scope

实现初始三个触发器：

| Trigger ID | 检测条件 |
|------------|---------|
| TRIGGER-001 | REQ body 含 API_KEY / SECRET / TOKEN / credential 关键词，但 .env.example 无对应变量 |
| TRIGGER-002 | inbox/for-pandas/ 收到 `type=major_decision_needed, blocking_reason: depends_on REQ-M` |
| TRIGGER-003 | inbox/for-pandas/ 收到 `type=major_decision_needed, blocking_reason: implementation requires changes outside REQ boundary` |

实现 `detect_major_decision <req_file>` 函数供 `pandas-heartbeat.sh` 调用：
- 返回 0 = 无触发，1 = 有触发（副作用：已调用 tg_decision）

# Out of Scope

- 追加新 trigger 的文档（在 PANDAS-ORCHESTRATION.md §5 中更新）
- Telegram 发送基础设施（由 REQ-024 提供）

# Acceptance Criteria

- `detect_major_decision tasks/features/REQ-XXX.md`（含 API_KEY）→ 触发 TRIGGER-001
- inbox 消息 `blocking_reason: depends_on REQ-M`（REQ-M 未完成）→ 触发 TRIGGER-002
- inbox 消息 `blocking_reason: implementation requires changes outside REQ boundary` → 触发 TRIGGER-003
- 无触发条件时 `detect_major_decision` 返回 0，不发送任何 Telegram 消息
- 新 trigger 可在 `detect_major_decision` 函数中以 `elif` 块扩展（无需修改调用方）
- `bash -n scripts/pandas-heartbeat.sh` 无语法错误
- `npm run release:audit` 通过
- `npm run req:check` 通过

# Test Case Design Notes

- TC-025-01: REQ body 含 `API_KEY` 且 .env.example 无对应行 → detect_major_decision 返回 1
- TC-025-02: REQ body 含 `API_KEY` 且 .env.example 已有对应行 → detect_major_decision 返回 0
- TC-025-03: inbox 消息 blocking_reason=depends_on → TRIGGER-002 触发
- TC-025-04: inbox 消息 blocking_reason=outside REQ boundary → TRIGGER-003 触发
- TC-025-05: 无触发条件 → detect_major_decision 返回 0，无 tg_decision 调用

# Agent Notes

- TRIGGER-001 检测：`grep -iE 'API_KEY|SECRET|TOKEN|credential' <req_file>` 提取候选词；
  再 `grep -F <candidate> .env.example` 检查是否已声明
- TRIGGER-002/003 检测：在 `inbox_read_pandas` 的 `major_decision_needed` handler 中调用
- `tg_decision` 由 REQ-024 提供；本 REQ 假设 REQ-024 已合并
- 扩展约定：每个新 trigger 在 `docs/PANDAS-ORCHESTRATION.md` §5 表格中新增一行
