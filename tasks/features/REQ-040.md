# REQ-040: Harness Compliance Eval Suite

## Goal

构建一个静态分析工具，用于检测每次 harness 规程（harness/*.md）、Agent 人设或
Runbook 变更后，Agent 的 workflow 遵从能力是否出现提升或退化。

核心问题：**修改了 harness 文档后，Agent 还能正确走完规定流程吗？**

## Background

当前 harness 开发循环没有可量化的反馈回路：修改规程后只能凭直觉判断好坏。
随着 REQ 数量增长，缺乏基准线会导致无法区分偶发失误与系统性退化。

## In Scope

- `eval/fixtures/` — 3 个 hello-world 级 fixture REQ（覆盖 tc_policy 三条路径）
- `scripts/eval-compliance.sh` — 静态 git log 解析，检查 3 步合规性
- `eval/baseline.json` — 基准合规率存储
- `eval/last-run.md` — 每次 eval 的 per-step 报告
- `npm run eval:compliance` — 主入口，支持 `--update-baseline` flag
- L1 单元测试（mock git log，不调用真实 Claude）

## Out of Scope

- 现场重新调用 Claude（live invocation）
- 公共 benchmark（GAIA、SWE-bench）
- 任务质量评估（fixture 只测流程，不测输出正确性）
- CI 自动触发（手动运行，不作为 PR gate）

## Compliance Steps Checked (per fixture branch)

| Step | Check Method | Pass Condition |
|------|-------------|----------------|
| ① claim | `git log -p feat/REQ-E00N` | commit with `owner: unassigned→claude_code` + `status: →in_progress` |
| ② pre-commit | commits after claim | ≥1 non-claim commit exists after claim commit |
| ③ review | latest REQ frontmatter | `status: review` in HEAD version of REQ file |

## Fixture REQ Design

```
eval/fixtures/
  REQ-E001.md   # tc_policy=exempt  — direct ready→in_progress path
  REQ-E002.md   # tc_policy=required — test_designed→in_progress path
  REQ-E003.md   # blocked scenario   — blocked→unblocked→in_progress path
```

Each fixture task is hello-world level (e.g., "create a shell script that prints hello").
This decouples task quality from harness compliance measurement. Agent completes in 5–10 min.
Fixtures do not change with codebase evolution.

## Acceptance Criteria (Detailed)

1. `npm run eval:compliance` 分析 `feat/REQ-E0*` branches：
   - 全部 9 步通过时 exit 0
   - 任一步骤失败时 exit 1，report 中标注具体失败步骤（✗）
   - compliance vs `eval/baseline.json` 下降 >10% 时 exit 1 并输出 `REGRESSION DETECTED`

2. `eval/last-run.md` 包含：
   - 每个 fixture 的 per-step 结果表格（✓ / ✗ / - ）
   - 总 compliance 百分比（通过步骤数 / 总步骤数）
   - 与 baseline 对比的 delta 值

3. `npm run eval:compliance -- --update-baseline`
   将当前结果写入 `eval/baseline.json`，exit 0（npm 需要 `--` 才能将 flag 传入脚本）

4. Graceful handling：
   - `feat/REQ-E00N` branch 不存在 → 该 task 标记 "not_run"，不计入 pass/fail
   - 至少 1 个 branch 存在时输出有效报告；否则 exit 1 with "no eval sessions found"

5. 全部通过 `npm run req:check && npm run build && npm test`

### Implementation (2026-03-26)

**Files created:**
- `src/eval/compliance.ts` — Core TypeScript module with pure functions (`evaluateFixture`, `runEval`, `renderReport`) + CLI entry point. Uses dependency injection (`getData` callback) for full testability without real git calls.
- `eval/fixtures/REQ-E001.md` — tc_policy=exempt fixture (direct ready→in_progress path)
- `eval/fixtures/REQ-E002.md` — tc_policy=required fixture (test_designed→in_progress path)
- `eval/fixtures/REQ-E003.md` — blocked scenario fixture
- `eval/baseline.json` — initial `{}` (no baseline until first `--update-baseline` run)
- `test/eval-compliance.test.ts` — 11 tests covering TC-040-01..05 with mock BranchData

**`package.json`:** added `eval:compliance` script pointing to `node --import tsx src/eval/compliance.ts`; baseline update requires `npm run eval:compliance -- --update-baseline` (double `--` required so npm passes the flag to the script, not treating it as an npm config option)

**Compliance detection:**
- Step ① claim: regex matches both git diff format (`+owner: claude_code`, `+status: in_progress`) and inline `→` notation
- Step ② pre-commit: splits git log by commit SHA boundaries, checks ≥1 newer commit exists after claim commit
- Step ③ review: checks HEAD REQ file content for `status: review`

**Graceful handling:** branch not found → `notRun=true`, excluded from pass/fail count; all branches missing → exit 1 "no eval sessions found"

**Regression gate:** compliance drop >10% vs baseline triggers exit 1 + "REGRESSION DETECTED" in report

**Quality gate:** `npm run release:audit && npm run build && npm test` all pass (276 tests)
