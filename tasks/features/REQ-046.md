# Goal

将 `scripts/harness.sh`（754 行）迁移至 `scripts/harness.ts`，提供类型安全的 Harness CLI 入口。

REQ-043~045 完成后，各心跳脚本可从 `child_process.spawn("bash harness.sh")` 改为直接 import harness TS 函数，消除最后一层 bash 桥接。

## In Scope

### 子命令迁移

| 子命令 | bash 函数 | TS 对应 |
|--------|-----------|---------|
| `status` | `cmd_status` | `cmdStatus()` |
| `implement <REQ-N>` | `cmd_implement` | `cmdImplement(reqId)` |
| `bugfix <BUG-N>` | `cmd_bugfix` | `cmdBugfix(bugId)` |
| `fix-review <REQ-N>` | `cmd_fix_review` | `cmdFixReview(reqId)` |
| `tc-review <PR-N>` | `cmd_tc_review` | `cmdTcReview(prNumber)` |
| `worktree-setup <REQ-N>` | `cmd_worktree_setup` | `cmdWorktreeSetup(reqId)` |
| `worktree-clean <REQ-N>` | `cmd_worktree_clean` | `cmdWorktreeClean(reqId)` |

### 依赖关系

- 使用 REQ-042 `parseReqFrontmatter`、`patchReqFrontmatter` 取代 `get_field`
- 使用 REQ-042 `inboxWrite` 取代 inbox 相关 bash 函数
- `check_depends_done` 直接读取 REQ frontmatter，不依赖 bash

### CLI 入口

```typescript
// scripts/harness.ts
const [,, command, ...args] = process.argv;
switch (command) {
  case 'status': await cmdStatus(); break;
  case 'implement': await cmdImplement(args[0]); break;
  // ...
}
```

兼容现有调用：`node --import tsx scripts/harness.ts implement REQ-042`

## Out of Scope

- 修改任何 harness 业务逻辑
- 删除 `harness.sh`（Umbrella done 后统一删除）
- `cmd_runbook`（调用外部渲染脚本，可后续迁移）

1. `scripts/harness.ts` 存在，`node --import tsx scripts/harness.ts status` 可执行

3. `scripts/harness.sh` 文件头部添加 `# @deprecated` 注释
4. REQ-043~045 的心跳脚本更新为直接 import harness TS 函数（消除 bash subprocess）
5. `cmdStatus` 有单元测试：输出 claimable REQ 列表，覆盖有任务和无任务两种情况
6. `cmdImplement` 有单元测试：REQ frontmatter 状态流转正确，mock `git`/`gh` 调用
7. `check_depends_done` 逻辑有单元测试：所有 depends_on 为 done 时返回 true，否则 false
8. `npm run build` 通过；`npm test` 通过；`npm run release:audit` 通过

## Agent Notes

### 迁移后的心跳脚本更新

REQ-043~045 完成时，各心跳脚本调用 harness 的方式是 `child_process.spawn("bash harness.sh")`。本 REQ 完成后，更新为：

```typescript
import { cmdImplement, cmdFixReview } from './harness';
// 直接调用，无需 subprocess
await cmdImplement(reqId);
```

这是整个重构链的最后一步，消除了 TS → bash 的最后一层桥接。

### 优先级为 P2 的原因

REQ-043~045 完成后，系统已经全面迁移到 TS，bash subprocess 桥接虽然存在但已隔离在一处。REQ-046 带来更干净的架构，但不是 P1 紧急项。可以在 REQ-043~045 完成并验证系统稳定后再启动。
