# Goal

为 P2 Agentic Deploy 序列（REQ-009–011）提供工具集，使 LLM ReAct Agent 在每台 Pi 的差异化环境中能够自主选路、自主恢复，而不是在第一个非预期错误处卡死。工具集的质量上限决定了 ReAct 的选路能力上限。

# In Scope

工具分四类，具体工具清单在 ready 阶段与 Daniel + CodeX 确认：

**1. 诊断工具（Diagnostic）**
- 查询 OS 版本、CPU 架构（arm64 / armv7）
- 查询 Node.js / npm 版本（是否存在、版本号）
- 检查代理连通性（curl through mihomo，返回 latency 或具体错误）
- 检查 systemd linger 状态（`loginctl show-user` — 影响 --user service 开机自启）
- 检查磁盘空间、内存基线

**2. 安装工具（Install）**
- Node.js 多路径安装：nvm / nodesource apt / 官方 tarball（按环境自动选路）
- openclaw binary 安装：npm global / GitHub release tarball / 本地包
- npm 代理配置（`npm config set proxy`）

**3. 配置验证工具（Assert）**
- 每个 REQ 步骤的幂等断言：可重复运行，已完成则跳过，未完成则返回差异
- openclaw.json 关键字段存在性检查
- .env 必填字段完整性检查
- systemd user service active 状态验证

**4. 错误翻译层（Error Translation）**
- 将 raw stderr（如 npm ERR! peer dep conflict）转为结构化 JSON，供 LLM 推理
- 将 curl / wget 网络错误归类（DNS 失败 / 超时 / 代理拒绝 / TLS 错误）
- 将 systemctl 失败日志（journalctl 最近 N 行）附加到工具返回值

# Out of Scope

- MCP server 的具体协议实现（是否走 MCP 或直接 CLI tool call，在 ready 阶段确认）
- 工具集覆盖 P2 以外的部署步骤（REQ-006/007/008 属于 auto-mihomo 和 Tailscale 范围）
- 工具集的 UI 或可视化

# Acceptance Criteria

- 工具集覆盖 REQ-009 / REQ-010 / REQ-011 的全部操作路径
- 每个安装类工具：成功时返回安装路径和版本号；失败时返回分类后的错误原因（非 raw exit code）
- 每个断言类工具：幂等，可重复运行，输出"已满足 / 未满足 + 差异描述"二态结果
- LLM（在测试场景中）在至少 2 种典型失败模式下能根据工具返回自主切换到备选路径
- `npm run req:check` 通过

# Test Case Design Notes

需要覆盖的场景（典型失败模式，具体 TC 在 ready 阶段设计）：
- TC: Node.js 未安装 → 诊断工具正确识别，LLM 触发 nvm 安装路径
- TC: npm 无代理配置 → 安装失败，错误翻译层返回"代理未配置"，LLM 触发代理设置步骤
- TC: systemd linger 未启用 → 断言工具检测到，LLM 执行 `loginctl enable-linger`
- TC: openclaw.json 路径错误 → 断言工具返回"文件不存在"，LLM 重新写入正确路径
- TC: 工具幂等性 — 在已完成环境中重跑全部断言，均返回"已满足"且无副作用

# Agent Notes

- 本 REQ 设计可与 REQ-009–011 并行进行，工具集在实现阶段由 REQ-009–011 的 Agent 使用
- 工具交付形式（MCP server / Claude Code custom tools / shell scripts + JSON output）在 ready 阶段与 Daniel + CodeX 确认；当前不预设实现方式
- 错误翻译层是最高杠杆工具：LLM 选路质量 = 工具返回信息质量，raw stderr 不可推理
- Pi 硬件差异矩阵（至少覆盖）：Pi 4 armv8 / Pi 3 armv7，Raspberry Pi OS Bookworm / Bullseye
