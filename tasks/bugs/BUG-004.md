# 现象描述

`requirement-standard.md v0.4`（REQ-029，2026-03-17）新增了 `review_ready` 状态，定义为：

> Human-to-Pipeline 交接点；Daniel 设置，Pandas 扫描并原子 commit 转为 `req_review`（state-as-lock，防重复认领）

但 `scripts/pandas-heartbeat.sh` 的 `auto_claim()` 函数完全没有处理 `review_ready`：

```bash
# 当前实现只检查 test_designed 和 ready
if [[ "$status" == "test_designed" && "$owner" == "unassigned" ]]; then
  claimable=true
elif [[ "$status" == "ready" && ... ]]; then
  claimable=true
fi
```

`harness.sh` 同样没有对应逻辑。

# 预期行为

Pandas 心跳时：
1. 扫描 `tasks/features/` 中所有 `status: review_ready` 的 REQ
2. 对每个匹配项执行原子 commit：`status → req_review`，`owner → huahua`
3. commit message 使用 `{REQ_ENTRY_HANDOFF_COMMIT}` 模式（`handoff({REQ_ID}): pandas -> huahua req_review`）
4. 写入 `inbox/for-huahua/` 需求评审 packet

防重复认领（state-as-lock）：原子 commit 保证同一 REQ 不被并发心跳双重认领。

# 复现步骤

1. 将任意 REQ 设置为 `status: review_ready`，`owner: unassigned`
2. 等待 Pandas 心跳（或手动触发 `APP_COMMAND=pandas-heartbeat`）
3. 观察：REQ 状态不变，未转为 `req_review`，Huahua inbox 无消息

# 根因分析

REQ-029 在 2026-03-17 更新了 `requirement-standard.md` 引入 `review_ready` 状态，
但对应的 `pandas-heartbeat.sh` 实现未同步更新。
规程先行、实现滞后导致状态机行为与文档不一致。

# 修复方案

在 `pandas-heartbeat.sh` 新增 `claim_review_ready()` 函数：

1. 扫描 `tasks/features/` 中 `status: review_ready` 且 `owner: unassigned` 的 REQ
2. 原子 commit 更新 frontmatter：`status → req_review`，`owner → huahua`
3. 写 `inbox/for-huahua/` packet（type: `req_review`，附 REQ 内容）
4. 在主 heartbeat 循环中调用（inbox 处理之后、auto_claim 之前）

同步更新 `harness.sh` 如有相关入口。
