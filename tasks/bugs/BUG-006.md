# 现象描述

通过 harness 自动化流程完成的 code review（`huahua-heartbeat.sh` 处理 `code_review` 消息），
往往一轮即 APPROVED，而手动在桌面用 Codex 跑相同 PR 时通常会发现多轮问题。

经代码审查确认，根因是多个设计缺陷的叠加：

1. **Huahua 身份未加载**：`huahua-heartbeat.sh` 调用 `claude --dangerously-skip-permissions -p`（one-shot print 模式），没有加载 `workspace-huahua/SOUL.md`、`IDENTITY.md`、`AGENTS.md`。Huahua 的"基于证据的怀疑主义"审查人格完全缺失。`review-standard.md` 文档里标注的"CodeX + GH LLM Issue Orchestrator skill"也没有实际注入

2. **只有 diff，没有代码上下文**：prompt 传入的是预先抓取的 `pr_diff` 字符串，没有要求 Huahua 先 `gh pr checkout ${pr_number}` 拿完整代码。无法读取被修改文件的完整上下文、相关测试文件、被测函数的调用方

3. **Acceptance Criteria 逐条核对未被强制**：prompt 要求"Contractual consistency"，但没有明确要求逐条找到每个 AC 对应的实现。如果某个 AC 在 diff 里没有明显对应，模型倾向于"合理假设"而不是标记 `[BLOCK]`

4. **verdict 只有两档（APPROVED / NEEDS_CHANGES），偏向快速收敛**：structured JSON output 约束导致模型在没有发现明显 `[BLOCK]` 时直接输出 `APPROVED`，缺少"需要更多证据"的中间状态，产生隐性的"尽快结束"压力

# 预期行为

- Huahua 执行 code review 时，应在完整 agent session 中运行，SOUL.md 和 IDENTITY.md 生效
- review 必须先 `gh pr checkout` 或等效操作，获取完整代码，不能只看 diff
- 每个 Acceptance Criterion 必须找到对应实现，否则标 `[BLOCK]`；这应是 prompt 的硬性步骤
- verdict 应支持"需要进一步检查"状态，不强迫在证据不足时二选一

# 根因分析

- `huahua-heartbeat.sh` 中 `code_review` handler 使用 `claude -p` one-shot，这是 bash 脚本对 LLM 的最简单调用方式，但丢失了 agent session 的所有语境
- `review-standard.md` 中的 skill 引用（CodeX + GH LLM Issue Orchestrator）是设计意图，未在 bash 实现中落地
- REQ-045（huahua-heartbeat.ts 迁移）和 REQ-043（pandas-heartbeat.ts 迁移）完成后，review 应改为通过 `sessions_spawn` 启动完整 ACP session，注入 `workspace-huahua/` 上下文

# 修复方案

## 短期修复（不依赖 REQ-041 重构，可立即实施）

在 `huahua-heartbeat.sh` `code_review` handler 的 prompt 中：

1. **Step 0 强制 checkout**：在 Steps 最前面加入 `gh pr checkout ${pr_number}` 作为必须执行的第一步
2. **AC 逐条核对**：增加明确步骤："对照 REQ 的每一条 Acceptance Criterion，在代码中找到对应实现。找不到对应实现的 AC 必须标记 `[BLOCK]: AC not implemented`"
3. **删除 `--output-format json --json-schema`**：允许 Claude 自由输出完整分析过程，最后输出结构化结论；或改为 `--output-format json` 但不约束 schema 让模型先输出 findings 再给 verdict

## 长期修复（配合 REQ-045）

`huahua-heartbeat.ts` 中 `code_review` handler 改为通过 `sessions_spawn` 启动完整 ACP session：
- `cwd` 指向 `workspace-huahua/`，使 Huahua SOUL.md/IDENTITY.md 生效
- verdict 扩展为三档：`APPROVED` / `NEEDS_CHANGES` / `NEEDS_MORE_EVIDENCE`
- `NEEDS_MORE_EVIDENCE` 时 Huahua 补充一轮证据收集后再出结论

# 影响范围

- 所有通过 harness 自动化流程合入的 PR，review 质量均受此影响
- 当前已合入的 PR（REQ-033 ~ REQ-040 等）存在 review 深度不足的风险，建议在重要 PR 上补一轮手动 review

# 验收标准
