# 现象描述

`requirement-standard.md` 的 TC 生命周期章节未定义 TC `status` 应在何时更新为 `done`。

结果是 TC 文件在实现完成后仍保留 `status: ready`（或 `test_designed`）等中间状态，
既未在实现验收通过时标记为 `done`，也未在归档时更正。

示例（实际存在的归档文件）：
```yaml
# tasks/archive/done/TC-028-01.md
tc_id: TC-028-01
status: ready   ← 应为 done
```

# 预期行为

TC `status` 应在**对应 REQ 的实现被验证完成**时（即 Menglan 实现通过、REQ 从 `in_progress` 转为 `review`，
或 Huahua 批准合并时）更新为 `done`。

归档（`mv` 到 `tasks/archive/done/`）是独立的文件管理操作，不是 status 变更的触发点。

```
TC 生命周期:
  ready / test_designed
    → (关联 REQ 实现完成 + 验证通过)
    → status: done
    → (post-merge 归档)
    → mv 到 tasks/archive/done/（status 保持 done）
```

如果 TC 事后发现问题，应通过新 BUG 跟踪，而不是将已 `done` 的 TC 回退状态。

# 根因分析

`requirement-standard.md` 的 TC 章节未明确 `done` 的触发条件和责任方。
§9.3 归档步骤只规定了文件移动，未要求在实现完成时同步 TC 状态，
导致实现者和自动化脚本都缺少明确的 TC `done` 动作指引。

# 修复方案

1. `requirement-standard.md` TC 生命周期章节补充：关联 REQ 实现验证通过时，责任方（Menglan 或 Pandas）将 TC `status` 更新为 `done`
2. §9.3 归档步骤说明：归档时 TC `status` 应已为 `done`；若不是，归档前先补更新
3. 对已归档但 `status` 不正确的历史 TC 做一次性修复
