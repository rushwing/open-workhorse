# 现象描述

`requirement-standard.md §9.3` 的归档步骤只要求移动文件：

```
- 将 tasks/test-cases/TC-xxx.md 同步移到 tasks/archive/done/
```

但没有要求在移动前将 TC 文件的 `status` 字段更新为 `done`。
结果是归档后的 TC 文件仍保留 `status: ready`（或 `test_designed`）等中间状态，
与文件位于 `archive/done/` 的事实语义不一致。

示例（实际存在的归档文件）：
```yaml
# tasks/archive/done/TC-028-01.md
tc_id: TC-028-01
status: ready   ← 应为 done
```

# 预期行为

§9.3 归档步骤应在移动 TC 文件前（或移动时）将 `status` 字段更新为 `done`：

```
- 将 TC status 更新为 done
- 将 tasks/test-cases/TC-xxx.md 移到 tasks/archive/done/
```

对应实现中，归档 commit 应包含 `status: done` 的 frontmatter 修改。

# 根因分析

§9.3 是从单 Agent 模式继承的文本，当时隐含由实现者在归档前手动完成状态同步。
多 Agent + 自动化归档设计后，未明确规定 TC 状态的更新责任。

# 修复方案

1. `requirement-standard.md §9.3` 补充：归档前将关联 TC 的 `status` 更新为 `done`
2. 实现层（`pandas-heartbeat.sh` 或归档脚本）在 `mv` 前执行 `sed` 更新 TC frontmatter
3. 对已归档的 status 不正确的 TC 文件做一次性修复（可在本 Bug 修复 PR 中附带）

# 回归验证

- [ ] 归档后 `tasks/archive/done/TC-xxx.md` 的 `status` 字段为 `done`
- [ ] 已有历史归档文件中 status 不正确的 TC 已修正
