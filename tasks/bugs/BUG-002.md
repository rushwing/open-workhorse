# 现象描述

当前规程对 `impl_bug` 的触发前置状态存在交叉冲突：

- [harness/bug-standard.md](../../harness/bug-standard.md) §2.4 将 `impl_bug` 定义为 `PR code review 阶段` 发现问题，但对应的 REQ 状态写成 `in_progress -> blocked`，unblock 后恢复到 `in_progress`
- [docs/req-flow.html](../../docs/req-flow.html) 的泳道图与 `REVIEW REJECTION` 文案更容易被理解为 `review` 阶段触发 `impl_bug`，修复后回到 `review`

这会让 Human 或 LLM 在同一问题上得到两套不同结论，已实际影响 Pandas 对 fail fallback 场景的回答。

# 预期行为

`bug-standard.md` 作为唯一高置信事实源，明确 `impl_bug` 的：

- 触发时机
- 触发时 REQ 状态
- block / unblock 后恢复状态
- 与 `req-flow.html`、相关 TC 文案完全一致

修复后，任何基于仓库规程的 agent 都不应再需要“自行选边解释”。

# 复现步骤

1. 打开 [harness/bug-standard.md](../../harness/bug-standard.md) §2.4，查看 `impl_bug` 行
2. 打开 [docs/req-flow.html](../../docs/req-flow.html)，查看 `impl_bug` 所在泳道和 `REVIEW REJECTION` 附近标签
3. 让一个 agent 基于这两份文档回答 “impl_bug 触发时 REQ 处于什么状态、关闭后恢复到什么状态”
4. 观察其容易给出 `review` 路径答案，而非明确指出与 bug-standard 的冲突

# 环境信息

- 分支：main
- 相关 commit：54c2c09

# 根因分析

PR #24 在升级 REQ v0.4 泳道图与需求规程时，没有把 `impl_bug` 分支语义与 `bug-standard.md` 做到逐项归一。
结果是：

- 事实源文本与辅助泳道图存在分支级语义漂移
- Human 阅读时更容易跟随泳道图
- LLM 也更容易直接采用泳道图语义而不显式升级冲突

# 修复方案

1. 以 `harness/bug-standard.md` 为唯一事实源，确定 `impl_bug` 的标准定义
2. 同步更新 `docs/req-flow.html` 中对应泳道、节点标签和说明文案
3. 同步检查并更新引用该分支语义的 REQ / TC 文案，避免继续传播旧口径
4. 在相关 TCs 中加入对关键泳道语义的校验，而不只检查状态关键字存在

# 回归验证

- [ ] `bug-standard.md` 与 `req-flow.html` 对 `impl_bug` 的触发状态、恢复状态表述一致
- [ ] 相关 REQ / TC 文案不再出现冲突口径
- [ ] 让 Pandas 基于仓库规程回答 `impl_bug` fail fallback 场景时，不再需要自行选边解释
