# Scenario

ci_bug 从 open 推进到 closed 的完整主路径。ci_bug 发生在 CI GitHub Actions 报红阶段，此时关联 REQ 已处于 `review` 状态（PR 已提交，等待 CI 验证）。confirmed 由 pandas 自动执行（无需人工）；关联 REQ 从 review → blocked → 恢复 review。

per bug-standard.md §2.4 路由表：ci_bug 触发时 REQ 处于 `review`，blocked_from_status=review，unblock 后恢复 review，最终目标 done（Daniel merge 后）。

# Steps

1. 构造 bug fixture：`bug_type=ci_bug, status=open, related_req=[REQ-001]`；构造 REQ fixture：`status=review`（注意：不是 req_review，ci_bug 发生在 CI 阶段，REQ 已过 req_review）
2. 调用自动确认流程（模拟 pandas 触发）；断言 `bug.status=confirmed, bug.owner=pandas`
3. 断言 confirmed 转换由 pandas 自动执行，无需额外人工操作
4. 断言 `REQ-001.status=blocked, REQ-001.blocked_reason=bug_linked, REQ-001.blocked_from_status=review`
5. 调用 transition(confirmed → in_progress)；断言 `bug.owner=menglan`
6. 调用 transition(in_progress → fixed)；断言 `bug.status=fixed`
7. 调用 transition(fixed → regressing)；断言 `bug.status=regressing`
8. 调用 transition(regressing → closed)；断言 `bug.status=closed`
9. 断言 `REQ-001.status=review`（恢复 blocked_from_status=review，不是 req_review）
10. 断言 `REQ-001.blocked_reason` 字段为空或不存在

# Expected

- ci_bug 触发者=pandas，confirmed 由 pandas 自动执行（区别于其他 bug 类型的手动确认）
- fix 责任人=menglan，reviewer=huahua（符合 §2.4 路由表）
- REQ unblock 恢复至 `review`（blocked_from_status），而非 req_review
- blocked_reason 和 blocked_from_status 字段清空
