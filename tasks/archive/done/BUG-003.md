# 现象描述

`requirement-standard.md §9.3` 定义了 PR 合并后的三个归档步骤：

```
- 把需求项改为 done
- 将 REQ-xxx.md 移到 tasks/archive/done/
- 将关联 TC-xxx.md 同步移到 tasks/archive/done/
```

但规程未指定：

1. **责任方**：这三步由谁执行（Pandas？Menglan？Daniel？）
2. **触发机制**：Daniel 在 GitHub 合并 PR 后，没有任何回调或轮询机制通知 Pandas 执行这三步

`harness-index.md` 的自动化流程结束于：
```
Pandas 发 tg_pr_ready → Daniel [Merge] / [Hold]
```
Daniel merge 后，流程图无后续节点。Pandas 既没有 webhook，也没有心跳检查项来感知 "PR 已 merge，对应 REQ 待归档"。

# 预期行为

1. §9.3 明确写入：归档步骤由 **Pandas** 执行（与 Pandas 发出 `tg_pr_ready` 的角色一致）
2. 触发机制在 `harness-index.md` 自动化流程中显性化：心跳 S2 扫描 `tasks/features/` 中 `status: done` 的文件，通知 Daniel 确认后执行归档
3. `HEARTBEAT.md` S2（已写入）作为触发入口，与规程文本对齐

# 复现步骤

1. 打开 `harness/requirement-standard.md` §9.3 "合并后"
2. 检索全局：哪里定义了 §9.3 步骤的责任方和触发事件
3. 观察：无任何规程文件指定归档责任方，也没有触发机制

# 根因分析

§9.3 是从单 Agent 模式继承的文本（隐含由 claude_code 在实现后自行处理），升级为多 Agent 半自治循环（Pandas/Menglan/Huahua + Daniel HITL）后未同步更新：

- 单 Agent 模式下，claude_code 既实现又归档，合并即归档，触发隐含在会话上下文内
- 多 Agent + HITL 模式下，merge 动作由 Daniel 在 GitHub 执行，Pandas 无法感知，导致归档步骤成为"无主"动作

# 修复方案

1. **requirement-standard.md §9.3** — 补充责任方：

   ```
   合并后（由 Pandas 在心跳 S2 检测到 status:done 后执行）：
   - 将 tasks/features/REQ-xxx.md 移到 tasks/archive/done/
   - 将关联 tasks/test-cases/TC-xxx.md 同步移到 tasks/archive/done/
   （pending_bugs 必须为空；§6.2 的 Bug clean 门控在 merge 时已满足）
   ```

2. **requirement-standard.md §6.2** — `review → done` 转换说明中补充：done 状态由 Menglan 在 PR 提交时预写（`status: done`），或由 Pandas 在 merge 确认后回写

3. **harness-index.md 自动化流程** — 在 `tg_pr_ready → Daniel [Merge]` 之后增加节点：

   ```
   Daniel merge PR
     └─▶ Pandas 心跳 S2：扫到 status:done → 发 Telegram 归档通知
             └─▶ Daniel 确认 → Pandas 执行归档（mv REQ + TC → archive/done/）
   ```

4. **新增 TC** — 验证 §9.3 三步的责任方文本、harness-index 流程图节点均存在
